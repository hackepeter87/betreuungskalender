# ADR 0001: Post-1.0 authentication architecture options

## Status

Proposed for post-`v1.0.0` planning.

## Context

The `v1.0.0` release line secures production deployment through an external TLS
reverse proxy, oauth2-proxy, and trusted identity headers. This keeps the app
small and makes OIDC deployment possible without adding native OIDC code before
the stable release.

Post-`v1.0.0` work will introduce controlled multi-user collaboration:

- stable OIDC users,
- claim-based authorization,
- actor identity in the audit log,
- personal calendar feeds, and
- revocable feed tokens.

Those requirements need an explicit authentication architecture decision before
implementation begins.

## Options

### Option A: oauth2-proxy with trusted headers

Keep oauth2-proxy as the OIDC client. The app trusts identity and claim headers
only when direct app access is blocked by container networking, firewall rules,
and reverse-proxy configuration.

Pros:

- operationally simple for the current deployment model,
- no native OIDC dependency in the app,
- keeps login/session handling outside the app, and
- fits the single-port Compose topology.

Cons:

- app security depends on strict network isolation,
- claim/header mapping must be documented and tested carefully,
- direct app exposure is unsafe when trusted-header auth is enabled, and
- calendar feed tokens still need an app-native token model.

### Option B: native OIDC inside the app

The app becomes the OIDC relying party and validates OIDC flows and tokens
itself.

Pros:

- clearer end-to-end ownership of user identity and claims,
- less reliance on trusted forwarding headers,
- easier to reason about user sessions inside the app, and
- may simplify future role diagnostics.

Cons:

- larger security-sensitive implementation,
- more configuration surface and callback/session handling,
- higher testing burden, and
- not needed to complete the stable `v1.0.0` release.

### Option C: hybrid

Keep oauth2-proxy for browser authentication, but add app-native concepts for
stable users, claim-derived permissions, audit actors, and revocable feed
tokens. Revisit native OIDC only if trusted-header deployment becomes too
limiting.

Pros:

- preserves the current production boundary,
- enables multi-user data modeling without replacing deployment auth,
- supports revocable calendar feed tokens independently from browser sessions,
- keeps native OIDC as a later decision, not a prerequisite.

Cons:

- still depends on safe proxy/header configuration,
- requires clear separation between browser identity and feed-token identity,
- may need migration if native OIDC is adopted later.

## Evaluation criteria

- OIDC users map to stable internal identities.
- Permissions are derived from configurable claims.
- Audit log entries record a reliable authenticated actor.
- Personal iCalendar feeds can use revocable, per-user tokens.
- Operators can deploy and debug the system safely.
- The app remains secure if a deployment accidentally exposes the wrong port.
- Existing single-user installations migrate safely.

## Initial direction

Use the hybrid approach for the first post-`v1.0.0` multi-user work:

1. keep oauth2-proxy and trusted headers for browser authentication;
2. add stable internal user records from configured OIDC-derived headers;
3. derive permissions from configurable claim/group headers;
4. enforce authorization server-side;
5. record authenticated actors in audit history; and
6. implement revocable feed tokens as scoped app-native credentials.

Do not implement native OIDC before the `v1.0.0` stable release. Revisit native
OIDC after the multi-user and calendar-feed requirements are clearer in real
operation.
