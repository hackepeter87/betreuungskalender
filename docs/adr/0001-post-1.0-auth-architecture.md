# ADR 0001: Native OIDC authentication target architecture

## Status

Accepted for `v1.4.0`.

This ADR supersedes the earlier post-`v1.0.0` direction that selected a
hybrid-first trusted-header model. The hybrid model was correct for `v1.1.0`
through `v1.3.0`: it enabled stable app users, claim authorization, actor
metadata, and personal calendar feeds without replacing oauth2-proxy during the
stable release rollout.

For `v1.4.0`, native OIDC becomes the target architecture. oauth2-proxy and
trusted proxy headers remain supported only as a transition and rollback mode
until native OIDC has been verified in a live deployment.

## Context

The current production authentication model is:

```text
Browser -> TLS reverse proxy -> oauth2-proxy -> Betreuungskalender -> SQLite
```

Betreuungskalender trusts configured identity and group headers only when
`TRUST_PROXY_AUTH=true`. This model is operationally workable, but the app's
security then depends on every deployment preserving a strict network boundary:
untrusted clients must never be able to reach the app service directly.

The app now has stable internal `app_users`, claim-derived roles, actor metadata,
and revocable personal calendar feed tokens. Those features make native session
and claim ownership practical. The next security milestone is to remove the
normal production dependency on trusted browser identity headers while keeping a
safe migration path.

## Decision

Betreuungskalender will implement native OIDC as a separate authentication mode:

```dotenv
AUTH_MODE=trusted-proxy|native-oidc|local
```

The target production mode is `AUTH_MODE=native-oidc`.

Native OIDC will use Authorization Code flow with PKCE. The app will be the OIDC
relying party, using an established OIDC library such as `openid-client`.
OIDC flow, token, issuer, audience, nonce, and expiry validation must not be
hand-rolled.

Browser endpoints:

- `GET /auth/login`
- `GET /auth/callback`
- `POST /auth/logout` or `GET /auth/logout` for browser navigation

The browser will receive only an opaque session cookie. ID tokens, access
tokens, refresh tokens, nonce values, and PKCE verifiers must never be stored in
`localStorage`, `sessionStorage`, frontend state, URLs after callback handling,
or other browser-readable storage.

Sessions will be server-side and stored in SQLite. The browser cookie will be:

- `HttpOnly`
- `Secure` in production
- `SameSite=Lax`
- scoped to the app path/domain
- opaque and high entropy

Login state, nonce, and PKCE verifier data will be short-lived, single-use, and
server-side. Callback handling must delete or invalidate consumed state.

## Authorization Model

Native OIDC claims map into the existing `app_users` and role model. The role
ordering remains:

```text
admin > parent > readonly
```

The configured group variables remain the public authorization interface:

```dotenv
OIDC_GROUPS_CLAIM=groups
OIDC_ADMIN_GROUP=/betreuungskalender/admins
OIDC_PARENT_GROUP=/betreuungskalender/parents
OIDC_READONLY_GROUP=/betreuungskalender/readers
OIDC_REQUIRE_ROLE_CLAIM=true
```

For native OIDC production deployments, `OIDC_REQUIRE_ROLE_CLAIM=true` is the
secure default. A user with a valid OIDC login but no matching configured group
must be authenticated but unauthorized for protected app access.

The stable internal user ID remains derived from the OIDC subject. Display name
and email are profile metadata only and must not be used as stable identity
keys.

## Transition and Rollback

`trusted-proxy` remains supported during the migration window. It is not removed
in `v1.4.0`.

The supported rollout sequence is:

1. Keep the current oauth2-proxy deployment as the known-good production path.
2. Add native OIDC configuration and deploy it first in staging or a controlled
   live maintenance window.
3. Verify login, callback, role mapping, logout, session expiry, and audit actor
   attribution.
4. Keep rollback simple: switch back to `AUTH_MODE=trusted-proxy` and the
   existing oauth2-proxy Compose topology if native OIDC fails validation.
5. Only after successful live operation decide whether trusted-proxy auth should
   be deprecated in a later release.

This avoids a risky Big Bang replacement.

## Required Security Properties

- Authorization Code + PKCE is required for browser login.
- State and nonce are validated for every callback.
- Issuer, audience/client ID, token signature, expiry, and supported algorithms
  are validated by the OIDC library.
- Tokens are never exposed to frontend JavaScript.
- Session identifiers are opaque, high entropy, server-side revocable, and
  rotated on login.
- Logout invalidates the local server-side session.
- Log output must not contain tokens, authorization codes, session cookies,
  state, nonce, PKCE verifier values, raw claims, or sensitive identity headers.
- Readonly users cannot write.
- Users without a matching role claim receive `403`.
- `trusted-proxy` mode continues to reject direct untrusted headers unless the
  deployment explicitly enables that transition mode.

## Public Configuration Interface

Native OIDC introduces:

```dotenv
AUTH_MODE=native-oidc
OIDC_ISSUER_URL=
OIDC_CLIENT_ID=
OIDC_CLIENT_SECRET=
OIDC_REDIRECT_URI=
OIDC_SCOPES=openid email profile
OIDC_GROUPS_CLAIM=groups
SESSION_COOKIE_NAME=betreuungskalender_session
SESSION_TTL_SECONDS=
```

Existing group and role variables remain supported:

```dotenv
OIDC_ADMIN_GROUP=/betreuungskalender/admins
OIDC_PARENT_GROUP=/betreuungskalender/parents
OIDC_READONLY_GROUP=/betreuungskalender/readers
OIDC_REQUIRE_ROLE_CLAIM=true
```

`/api/session` remains the frontend's central auth source and keeps its
compatible shape:

```ts
{
  authRequired: boolean;
  authenticated: boolean;
  user?: {
    id: string;
    displayName: string;
    role: "admin" | "parent" | "readonly";
    email?: string;
  };
  loginUrl?: string;
  logoutUrl?: string;
}
```

## Implementation Plan

Implementation is intentionally split into reviewable packages:

- #89 implements OIDC client discovery, login redirect, callback validation,
  Authorization Code + PKCE, state, and nonce handling.
- #90 implements server-side session storage, cookie lifecycle, session expiry,
  and logout invalidation.
- #91 implements `AUTH_MODE` switching and trusted-proxy compatibility.
- #84 ports claim-based role mapping from trusted headers to native OIDC claims.
- #85 updates frontend login/logout UX while keeping `/api/session` as the UI
  contract.
- #92 documents native OIDC fresh install with Keycloak and Podman.
- #94 documents oauth2-proxy to native OIDC migration and rollback.
- #93, #95, #96, and #87 provide security regression coverage and acceptance
  gates.
- #88 handles release hardening and the later trusted-proxy deprecation decision
  after native OIDC has been verified in operation.

Issue #83 remains a parent tracking issue for the native
login/callback/logout/session model and should not be implemented as one large
PR.

## Consequences

Positive:

- The normal production security boundary no longer depends on trusting browser
  identity headers from a proxy.
- The app owns session lifecycle, logout semantics, and claim diagnostics.
- Role enforcement is easier to test against OIDC claims directly.
- Rollback remains available through the proven oauth2-proxy deployment.

Negative:

- The app gains security-sensitive OIDC and session code.
- More configuration is required for native deployments.
- SQLite must store session and login-state records safely.
- Test coverage must expand before any production rollout.

## Non-Goals

- Do not remove oauth2-proxy or trusted-proxy mode in `v1.4.0`.
- Do not implement CalDAV or calendar writeback as part of native OIDC.
- Do not introduce browser-visible tokens.
- Do not add a user-management UI as part of the initial native OIDC feature.
- Do not support multiple unrelated OIDC providers in the first implementation.

## Acceptance Gates

Before native OIDC can be considered releasable:

- `npm run lint`
- `npm run test`
- `npm run build`
- `npm run test:security-runtime`
- `npm run test:e2e`
- Container smoke with `Dockerfile.release`
- Native OIDC security tests for state, nonce, PKCE, issuer, audience, expiry,
  session cookies, logout, missing groups, readonly writes, and log redaction
- Manual Keycloak acceptance for admin, parent, readonly, and no-group users
- Documented rollback back to `AUTH_MODE=trusted-proxy`
