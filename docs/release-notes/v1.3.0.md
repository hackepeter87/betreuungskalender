# v1.3.0 - Personal calendar feed

## Summary

`v1.3.0` completes the personal calendar feed milestone. Signed-in users can
generate a protected iCalendar subscription URL for their own care entries and
subscribe to it from calendar applications.

The release also improves the recurring contact-rule workflow with a
calendar-style preview before generation, and carries forward the documented
GHCR release-image deployment path.

Native OIDC is not introduced by this release. The trusted-proxy/OIDC topology
from `v1.1.0` and `v1.2.0` remains the supported production authentication
path.

## Upgrade notes

- Use the matching `v1.3.0` release archive and checksum once the release is
  published.
- Back up the SQLite database and private deployment configuration before
  updating.
- Keep the existing `TRUST_PROXY_AUTH=true` deployment private behind the
  trusted proxy when using oauth2-proxy.
- Feed URLs are bearer secrets. Treat them like passwords and rotate or revoke
  them if they may have been shared unintentionally.

## Migration impact

Migration `008_calendar_feed_tokens` adds the `calendar_feed_tokens` table for
per-user feed token hashes. It does not change existing care entries, children,
audit rows, backups, exports, or OIDC user records.

The database stores only SHA-256 hashes of generated feed tokens. Existing rows
are not backfilled because feeds are opt-in and must be generated by each user.

As with every forward SQLite migration, downgrades are not supported. Restore
the pre-update SQLite backup together with the previous release directory if a
rollback is required.

## User-visible changes

- Settings now include a `Persönlicher Kalenderfeed` section.
- Users can generate, rotate, copy, and revoke their own `.ics` feed URL.
- The feed exports non-deleted and non-cancelled care entries created by the
  signed-in user.
- Recurring contact-rule generation now shows preview cards that visualize the
  affected week days and identify already existing occurrences before entries
  are generated.

## Security note

The calendar feed token grants access only to `/calendar/<token>.ics`; it does
not authenticate API requests. The feed deliberately excludes notes, evidence
references, trips, costs, audit metadata, deleted entries, and cancelled
entries.

Request logging redacts calendar feed token paths before writing request
metadata. The feed endpoint is rate-limited with `RATE_LIMIT_EXPORT_MAX` and
`RATE_LIMIT_WINDOW_MS`.

## Validation

Before publication, validate:

- `npm run lint`
- `npm run test`
- `npm run build`
- `npm run test:security-runtime`
- `npm run test:e2e`
- Browser smoke test for settings -> calendar feed creation and `.ics` fetch
- Docker release-image smoke test with `DOCKERFILE=Dockerfile.release`
- `npm run release:check`

After publication, download the release archive and checksum, verify the
SHA-256 value, and run the normal runtime verification:

```bash
npm run verify:runtime -- --expected-version 1.3.0
```
