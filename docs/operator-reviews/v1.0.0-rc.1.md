# v1.0.0-rc.1 operator documentation review

Date: 2026-06-29

Release: <https://github.com/hackepeter87/betreuungskalender/releases/tag/v1.0.0-rc.1>

## Scope

This review checked the operator-facing installation, backup, restore, update
dry-run, and rollback documentation against the published `v1.0.0-rc.1`
artifact. It used isolated temporary paths and fictional data only.

No production deployment was performed.

## Artifact and installation

- Downloaded `betreuungskalender-v1.0.0-rc.1.tar.gz`.
- Downloaded `betreuungskalender-v1.0.0-rc.1.tar.gz.sha256`.
- Verified the checksum with `shasum -a 256 -c`.
- Confirmed SHA-256:
  `0e6a2ccbf86f54d809710711464abf4d443270d43a61f9b6d02be4e1b98d517f`.
- Extracted the archive into an isolated runtime directory.
- Installed production dependencies with `npm ci --omit=dev`.

Result: passed with `0 vulnerabilities`.

## Runtime verification

The runtime was started from the extracted release archive with temporary
SQLite and backup paths. The following checks passed:

```bash
npm run healthcheck
npm run verify:runtime -- --expected-version 1.0.0-rc.1
```

The health endpoint reported version `1.0.0-rc.1`; readiness confirmed that the
database was reachable and migrations were applied.

## Backup and restore

A fictional child record named `Fiktives Betreiber-Testkind` was created
through the API. The documented backup and restore-check commands were then
tested with temporary paths:

```bash
npm run backup
npm run restore:check
```

Result: passed. The backup file was created in the configured backup directory,
and restore verification succeeded.

## Update dry-run

A synthetic Compose root was prepared with temporary `compose.yml`, `.env`,
`data/`, `backups/`, and `releases/` paths. Docker Compose was available
locally.

The first dry-run attempt intentionally exposed an operator-documentation
clarity issue: the updater refuses to target a release directory that already
exists. The update documentation now calls this out explicitly.

After removing the pre-existing target directory and pointing
`APP_RELEASE_DIR` at the currently installed runtime, the documented dry-run
flow passed:

```bash
node scripts/update.js \
  --root /tmp/betreuungskalender-operator-doc-review/compose-root \
  --version 1.0.0-rc.1 \
  --artifact /tmp/betreuungskalender-operator-doc-review/betreuungskalender-v1.0.0-rc.1.tar.gz \
  --checksum /tmp/betreuungskalender-operator-doc-review/betreuungskalender-v1.0.0-rc.1.tar.gz.sha256 \
  --dry-run
```

Result: passed. The updater reported `preflight_passed` and `dry_run_complete`.

## Documentation findings

- `docs/update.md` still used concrete `v0.5.0` and `0.6.0` examples in
  operator commands. Those examples now use reusable `vCURRENT` and `X.Y.Z`
  placeholders, with an explicit `1.0.0-rc.1` example note.
- `docs/update.md` did not explicitly state that the target
  `releases/vX.Y.Z/` directory must not already exist. It now does.
- `docs/deployment-container.md` blurred the local checkout `compose.yaml`
  path with the release-archive `deploy/compose.yml` path. It now describes
  both entry points separately.

## Rollback guidance

The rollback documentation was reviewed for reverse-migration ambiguity. It
correctly states that the updater never attempts reverse database migrations
and that rollback restores the prior runtime together with the matching SQLite
backup.

## Result

The published `v1.0.0-rc.1` artifact passed the operator documentation review.
No release-candidate blocker was found.

Remaining stable-release work should be limited to actual release-candidate
defects or final `1.0.0` publication tasks.
