# v1.2.0 - Multi-parent collaboration and audit history

## Summary

`v1.2.0` completes the multi-parent collaboration and audit-history milestone.
The application now stores stable creator and updater metadata across the main
domain records and surfaces useful "last changed by" context in the existing
lists.

This release builds on the trusted OIDC claim authorization introduced in
`v1.1.0`. It does not replace oauth2-proxy and does not introduce native OIDC
login.

## Upgrade notes

- Use the matching `v1.2.0` release archive and checksum once the release is
  published.
- The published release includes a GHCR runtime image digest asset. Prefer the
  immutable digest reference if deploying from GHCR; the archive-based update
  path remains the primary documented update path.
- Back up the SQLite database and private deployment configuration before
  updating.
- Keep the current trusted-proxy/OIDC deployment topology from `v1.1.0`.
- For first live rollout after `v1.1.0`, keep
  `OIDC_REQUIRE_ROLE_CLAIM=false` until Keycloak/oauth2-proxy group headers are
  confirmed. If group forwarding is already verified, `true` remains the
  stricter production setting.

## Migration impact

Migration `007_actor_metadata` adds `created_by` and `updated_by` metadata to
domain tables that previously did not store full actor attribution directly:

- children
- trips
- costs
- holiday periods
- contact patterns
- monthly closures

Care entries and unavailable periods already stored actor metadata and are
preserved. Existing rows are backfilled with the local development identity
`local-dev` when no more specific actor is available. Existing audit entries,
backups, exports, and user-facing domain data are not deleted.

As with every forward SQLite migration, downgrades are not supported. Restore
the pre-update SQLite backup together with the previous release directory if a
rollback is required.

## User-visible changes

- Care-entry lists show who last changed each entry and when.
- Child settings show the latest editor for each child record.
- Holiday, unavailable-period, and contact-date lists show last-change context
  without requiring a separate user-management screen.
- The audit log resolves internal user IDs to display names where possible.

## Security note

The release keeps the same security boundary as `v1.1.0`: when
`TRUST_PROXY_AUTH=true`, the app must remain private behind the trusted proxy.
Browsers and untrusted clients must not be able to reach the app service
directly because trusted identity and group headers are accepted from the proxy
boundary.

No OAuth tokens or session secrets are stored in the browser by this release.

## Validation

Before publication, validate:

- `npm run lint`
- `npm run test`
- `npm run build`
- `npm run release:check:strict`

After publication, download the release archive and checksum, verify the
SHA-256 value, and run the normal runtime verification:

```bash
npm run verify:runtime -- --expected-version 1.2.0
```
