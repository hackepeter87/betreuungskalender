# v1.0.0-rc.2 - OIDC deployment release candidate

## Release candidate goal

`v1.0.0-rc.2` is the second release candidate for the `1.0.0` stable release.
It exists because `v1.0.0-rc.1` was published before the supported single-port
OIDC deployment files and rootless deployment guidance were added.

This candidate creates a new verified artifact for operators who want the OIDC
Compose path before the final `1.0.0` release. It does not introduce product
feature scope, a database schema migration, or an API contract change.

## Included changes since v1.0.0-rc.1

- Adds `deploy/compose.oidc.yml` for a release archive deployment where only
  oauth2-proxy exposes a host port.
- Adds `deploy/.env.oidc.example` and `deploy/oauth2-proxy.cfg.example` as
  neutral, placeholder-only starting points.
- Keeps the app service private in OIDC mode with `expose: 3000` and no app
  host `ports:` mapping.
- Uses `http://betreuungskalender:3000` as the oauth2-proxy upstream on the
  private Compose network.
- Documents `APP_COMPOSE_FILE=compose.oidc.yml` for managed updates in OIDC
  deployments.
- Hardens rootless Podman guidance for bind addresses, external reverse
  proxies, oauth2-proxy config-file permissions, cookie secret generation,
  health checks, and troubleshooting.
- Removes deployment-specific real examples from generic documentation.
- Adds a post-`1.0.0` authentication architecture decision record for native
  OIDC, multi-parent collaboration, and protected calendar feeds.
- Extends release and update validation so the OIDC topology and archive paths
  are checked automatically.

## Important artifact note

The published `v1.0.0-rc.1` archive does not contain:

- `deploy/compose.oidc.yml`
- `deploy/.env.oidc.example`
- `deploy/oauth2-proxy.cfg.example`

Do not mix deployment files copied from `main` into an older release artifact as
the normal production path. Use a verified `v1.0.0-rc.2` or newer artifact so
the runtime, scripts, examples, and documentation are reviewed together.

## Upgrade from v1.0.0-rc.1

No database schema migration is introduced by `v1.0.0-rc.2`.

For Compose-based deployments, follow the documented update process:

1. Download `betreuungskalender-v1.0.0-rc.2.tar.gz` and the matching
   `.sha256` file from the release.
2. Verify the checksum before extraction or execution.
3. Review these release notes.
4. Run the updater with `--dry-run`.
5. Run the managed update only after the dry run succeeds.
6. Confirm `/api/health` and `/api/ready`.

For OIDC deployments, start from `deploy/.env.oidc.example` and
`deploy/oauth2-proxy.cfg.example`, replace all placeholders privately, and keep
the resulting `.env` and `oauth2-proxy.cfg` out of Git.

## Known limitations

- The application documents user-entered facts and does not provide legal
  advice or an authoritative legal assessment.
- Docker Compose is the supported automated update path.
- Direct Node.js/systemd deployments remain a documented manual fallback.
- The release does not publish a container image to GHCR.
- OIDC support is provided through the oauth2-proxy deployment topology; native
  in-app OIDC remains post-`1.0.0` work.
- Imported external calendars remain read-only and do not affect care reports,
  statistics, exports, or month closure.

## Release-candidate policy

After this release candidate, only defects that affect the existing release
candidate scope should block `1.0.0`. New product ideas should be scheduled
after the stable release unless they fix a clear RC blocker.
