# v1.0.0-rc.1

## Release candidate goal

`v1.0.0-rc.1` is the first release candidate for the `1.0.0` stable release.
It promotes the validated `v0.5.0` product and operations baseline without
introducing new product scope, database schema changes, or deployment changes.

The purpose of this candidate is to freeze the current feature set, validate the
release path, and leave only release-candidate defects between this version and
the stable `1.0.0` release.

## Included baseline

This release candidate includes the completed application foundation:

- SQLite/API-backed domain persistence for all current care documentation data.
- Legacy browser-data migration and conflict handling.
- Children, care entries, Soll-Ist contact rules, holidays, unavailable periods,
  trips, costs, monthly closure, data-quality checks, and audit logging.
- JSON import/export, CSV exports, neutral PDF reporting, and print view.
- Responsive desktop, iPhone, and iPad support with E2E coverage.
- German-first interface with language-pack structure and English resources.
- Read-only external holiday calendar import and calendar overlays.
- SQLite backup, restore verification, health, readiness, update, and rollback
  scripts.
- Compose-oriented runtime archive and release validation.
- Security headers, proxy-auth support, CORS controls, and central API rate
  limiting.

## Validation focus

The RC release path is expected to pass:

- `npm ci`
- `npm audit`
- `npm run release:check`
- `npm run lint`
- `npm run build`
- `npm test`
- `npm run test:e2e`
- `npm run test:security-runtime`
- release archive validation on the `v1.0.0-rc.1` tag
- container runtime validation on the `v1.0.0-rc.1` tag

The previously published `v0.5.0` artifact was manually smoke-tested outside
CI. That smoke test verified archive checksums, archive contents, clean runtime
startup, health/readiness/version reporting, synthetic SQLite backup and
restore, and update dry-run behavior.

## Upgrade from v0.5.0

No database schema migration is introduced by `v1.0.0-rc.1`.

For Compose-based deployments, follow the documented update process:

1. Download `betreuungskalender-v1.0.0-rc.1.tar.gz` and the matching
   `.sha256` file from the release.
2. Verify the checksum before extraction or execution.
3. Review these release notes.
4. Run the updater with `--dry-run`.
5. Run the managed update only after the dry run succeeds.
6. Confirm `/api/health` and `/api/ready`.

For direct Node.js/systemd deployments, follow the manual fallback procedure in
the update documentation: create and verify a SQLite backup, retain the prior
runtime, install the release candidate, start the service, verify health and
readiness, and restore both runtime and database if validation fails.

## Known limitations

- The application documents user-entered facts and does not provide legal
  advice or an authoritative legal assessment.
- Docker Compose is the supported automated update path.
- Direct Node.js/systemd deployments remain a documented manual fallback.
- The release does not publish a container image to GHCR.
- Imported external calendars remain read-only and do not affect care reports,
  statistics, exports, or month closure.

## Release-candidate policy

After this release candidate, only defects that affect the candidate's existing
scope should block `1.0.0`. New product ideas should be scheduled after the
stable release unless they fix a clear RC blocker.
