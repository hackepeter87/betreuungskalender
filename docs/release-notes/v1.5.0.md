# v1.5.0 - Contact rules and calendar sharing

## Summary

`v1.5.0` completes the contact-rule and calendar-sharing milestone. Recurring
care is now modeled as flexible calendar-like rules instead of a fixed 14-day
special case. Saving a rule immediately synchronizes planned calendar entries,
so the core workflow is visible without a separate generation step.

The release also introduces care parties as the domain model for responsible
caregivers. Authenticated users remain technical actors for auth and audit;
care parties describe who the children are with. This keeps solo self-management
simple while enabling optional shared operation later.

## Changes

- Added flexible weekly and monthly recurring contact rules with presets for
  common care arrangements, custom weekday selections, ordinal weekdays, and
  weekend patterns.
- Added calendar-style rule previews that show the real generated dates before
  saving.
- Added automatic synchronization from saved rules into planned `care_entries`
  for the configured planning horizon.
- Added single-occurrence versus series editing behavior so manually changed or
  cancelled generated entries are preserved.
- Added care parties with editable names and caregiver kinds such as father,
  mother, grandparent, foster caregiver, and other.
- Added optional OIDC user-to-care-party assignments for installations that
  want shared management.
- Added scoped iCalendar feeds:
  - `legacy` for existing pre-v1.5 tokens based on `created_by`
  - `all` for all visible active care entries
  - `party:<id>` for one selected care party
- Added end-to-end coverage for rule creation, calendar visibility, care-party
  assignment, and feed scope output.

## Upgrade notes

- This release adds SQLite migrations for flexible contact rules, care parties,
  contact-rule synchronization metadata, calendar feed scopes, and optional
  user-to-care-party assignments.
- Existing 14-day contact patterns are migrated into flexible contact rules
  where possible.
- Existing active care entries and contact rules without a responsible party
  are assigned to a neutral default care party named `Primary caregiver`.
- Existing calendar feed tokens remain valid as `legacy` tokens. Rotate feed
  tokens from the settings page to use `all` or care-party scoped feeds.
- Solo operation remains the default. Do not configure user-to-care-party
  assignments unless the installation should restrict non-admin users to their
  assigned care parties.
- As with every forward SQLite migration, downgrades are not supported. Restore
  the verified pre-update SQLite backup together with the previous release if a
  rollback is required.

## Security notes

- Calendar feed URLs are still bearer secrets. Store them only in trusted
  calendar clients and rotate them if exposed.
- Feed output excludes notes, evidence references, trips, costs, audit metadata,
  deleted entries, and cancelled entries.
- OIDC user-to-care-party assignments do not replace role authorization.
  `app_users.role` still determines admin, parent, and readonly permissions.
- When at least one care-party assignment exists, non-admin write operations for
  care entries and contact rules must use one of the caller's assigned care
  parties. Admin users stay unrestricted for repair and administration.
