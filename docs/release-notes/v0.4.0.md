# v0.4.0

## Highlights

This release adds typed language-pack infrastructure and provider-independent
holiday calendar import while strengthening reproducible release validation.

## Added

- German remains the default/fallback language, with an implemented English
  language pack, localized reports, and PDF vocabulary.
- Local `.ics` holiday calendar file import with named sources, configurable
  colors, show/hide controls, replacement, deletion, and read-only calendar
  overlays.
- External calendar sources and normalized events are included in backup and
  restore data. Database migration is automatic through the existing migration
  runner.
- Expanded unit, migration, API, and desktop/iPhone/iPad E2E coverage.

## Improved

- Release and deployment automation now supports reproducible validation,
  checksummed release artifacts, and container health validation.
- The completed UI-string migration provides consistent localized interface
  copy across the implemented language-pack scope.

## Security and dependencies

- Updated `dompurify` to remove the previously reported moderate dependency
  advisory.
- External calendar routes are rate-limited and imports enforce file, event,
  and field-size limits.

## Compatibility

- Existing SQLite databases upgrade automatically.
- Existing backups remain supported; new backups include external calendar
  sources and events.

## Upgrade notes

1. Create and verify a current backup before upgrading.
2. Run `npm ci`, `npm run lint`, `npm test`, and `npm run build`.
3. Start the app and allow the migration runner to apply database changes.

## Known limitations

- External calendars are imported from local `.ics` files only.
- Imported events are read-only and do not affect care reports, statistics,
  exports, or month closure.
- URL subscriptions, remote refresh, provider-specific integration, and
  CalDAV are not included.
