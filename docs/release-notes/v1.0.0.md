# v1.0.0 - Stable release

## Release goal

`v1.0.0` is the first stable release of Betreuungskalender. It promotes the
validated `v1.0.0-rc.2` line and includes the final PWA title/icon polish
required before marking the release stable.

The release remains a self-hosted documentation tool. It does not provide legal
advice, create an official record, or guarantee that generated material will be
accepted by a court, authority, lawyer, or other recipient.

## Included capabilities

- SQLite/API domain persistence for children, care entries, holidays, contact
  rules, unavailable periods, trips, costs, settings, monthly closings, and
  audit records.
- Responsive React frontend with desktop, iPhone, and iPad coverage.
- Calendar, agenda, entries, analytics, CSV/JSON export, PDF report, print
  view, documentation rules, backup/import, and legacy browser-data migration.
- Planned/actual contact schedule handling, holiday allocation, duty-related
  unavailability, cost and trip documentation, monthly closure, data-quality
  checks, soft deletion, and field-level audit logging.
- Local read-only `.ics` holiday calendar imports with isolated calendar
  overlays.
- German-first UI with language-pack structure and English catalog coverage.
- Production-oriented Fastify API with SQLite migrations, reverse-proxy
  authentication support, restrictive CORS, security headers, and central rate
  limiting.
- Docker Compose release archive deployment, managed updates, backup
  verification, readiness checks, runtime/database rollback, and rootless
  Podman-compatible OIDC deployment through oauth2-proxy.
- PWA metadata with the full app name, favicon, Apple touch icon, and
  installable app icons.

## Changes since v1.0.0-rc.2

- Sets the web manifest short name to `Betreuungskalender`.
- Adds repo-owned SVG, PNG, maskable, Apple touch, and favicon assets.
- Links browser and Apple icons from `index.html`.
- Updates the service worker app-shell cache for the new icon assets.
- Adds regression tests for PWA metadata and public icon presence.
- Updates `.env.example`, OIDC environment examples, README status, and
  deployment documentation to use the stable `1.0.0` release path.

## Deployment notes

For release-archive deployments, use the matching `v1.0.0` archive and checksum.
Do not mix files copied from `main` into an older release artifact as the normal
production path.

The standard Compose example uses:

```dotenv
APP_RELEASE_VERSION=1.0.0
APP_RELEASE_DIR=/opt/svc_betreuung/betreuungskalender/releases/v1.0.0
```

For internet-facing OIDC deployments, use `deploy/compose.oidc.yml` with
oauth2-proxy as the only service that publishes a host port. Keep edited `.env`
and `oauth2-proxy.cfg` files private and out of Git.

## Validation focus

The release should pass:

- `npm ci`
- `npm audit`
- `npm run release:check`
- `npm run lint`
- `npm run build`
- `npm test`
- `npm run test:e2e`
- `npm run test:security-runtime`
- release archive validation on the `v1.0.0` tag
- container runtime validation on the `v1.0.0` tag

## Known limitations

- The application documents user-entered facts and does not provide legal
  advice or an authoritative legal assessment.
- Docker Compose is the supported automated update path.
- Direct Node.js/systemd deployments remain a documented manual fallback.
- The release does not publish a container image to GHCR.
- OIDC support is provided through the oauth2-proxy deployment topology; native
  in-app OIDC remains post-`1.0.0` work.
- Imported external calendars remain read-only and do not affect care reports,
  statistics, exports, or month closure.
