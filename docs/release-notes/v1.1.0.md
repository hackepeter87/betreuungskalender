# v1.1.0 - OIDC claim authorization

## Summary

`v1.1.0` adds server-side authorization for deployments that use trusted
OIDC/oauth2-proxy headers. The API now maps the stable trusted OIDC subject to
an internal `app_users` record and derives permissions from configured group
claims.

This release keeps the existing single-user trusted-proxy rollout path
compatible while operators verify group headers.

## Upgrade notes

- Use the matching `v1.1.0` release archive and checksum once the release is
  published.
- Back up the SQLite database and the private deployment configuration before
  updating.
- Existing deployments must review and add the new `OIDC_*` variables from
  `.env.example` or `deploy/.env.oidc.example`.
- First live rollout instruction:

```dotenv
OIDC_REQUIRE_ROLE_CLAIM=false
```

Keep this value for the first rollout. Switch it to `true` only after
Keycloak/oauth2-proxy group headers are confirmed in the live deployment.

## Migration impact

Migration `006_oidc_users` creates the `app_users` table, indexes it by trusted
external subject and role, and seeds the local development identity
`local-dev` as an admin user.

Existing children, care entries, holidays, contact rules, costs, trips,
unavailable periods, audit entries, backups, and exports are not rewritten by
this migration. As with every forward SQLite migration, downgrades are not
supported; restore the pre-update SQLite backup if rollback is required.

## Configuration impact

Review these settings for protected deployments:

```dotenv
REQUIRE_AUTH=true
TRUST_PROXY_AUTH=true
AUTH_LOGOUT_URL=/oauth2/sign_out
OIDC_USER_ID_HEADER=x-auth-request-user
OIDC_EMAIL_HEADER=x-auth-request-email
OIDC_DISPLAY_NAME_HEADER=x-auth-request-preferred-username
OIDC_GROUPS_HEADER=x-forwarded-groups
OIDC_ADMIN_GROUP=/betreuungskalender/admins
OIDC_PARENT_GROUP=/betreuungskalender/parents
OIDC_READONLY_GROUP=/betreuungskalender/readers
OIDC_REQUIRE_ROLE_CLAIM=false
```

For the release Compose OIDC topology, oauth2-proxy forwards group claims to
the app upstream as `X-Forwarded-Groups`; configure
`OIDC_GROUPS_HEADER=x-forwarded-groups` and verify `/api/session` reports the
expected role before enabling strict role-claim enforcement.

The default compatibility mode grants parent permissions to authenticated proxy
users when no configured group claim matches. After group forwarding is
verified, set `OIDC_REQUIRE_ROLE_CLAIM=true` to reject users without a matching
admin, parent, or readonly group.

## Deployment checklist

1. Create and verify a SQLite backup.
2. Update the release archive path and version only after checksum validation.
3. Add or review the `OIDC_*` variables in the private `.env`.
4. Confirm oauth2-proxy forwards the configured user, email, display-name, and
   groups headers.
5. Confirm direct client access to the app service is blocked when
   `TRUST_PROXY_AUTH=true`.
6. Start with `OIDC_REQUIRE_ROLE_CLAIM=false`.
7. Verify `/api/session` shows the expected signed-in user.
8. Verify ordinary read/write behavior with a non-sensitive test record.
9. Verify admin-only operations only after admin group forwarding is confirmed.
10. Switch `OIDC_REQUIRE_ROLE_CLAIM=true` only after group headers are known to
    match the configured values.

## Rollback note

Use the managed update rollback path or restore the pre-update release
directory, `.env`, and SQLite backup together. A container-only rollback is not
enough after migration `006_oidc_users` has been applied.

## Security note

Trusted proxy headers are security-sensitive. When `TRUST_PROXY_AUTH=true`, the
app must not be directly reachable by browsers or untrusted clients. Keep the
app service private behind oauth2-proxy, strip incoming identity headers at the
public edge, and expose only the intended proxy endpoint.
